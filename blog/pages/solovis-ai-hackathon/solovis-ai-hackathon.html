The Hackathon

Solovis held an AI-First Hackathon on April 21st, 2026 — a day-long event designed to reinforce what it means to develop with AI as a foundational tool, not an afterthought.

The hackathon's core thesis was straightforward: "Build smarter. Move faster. Think differently." It wasn't just about shipping something in a few hours — it was about rewiring how we work. The skills and workflows practiced that day were meant to carry forward into future work. AI-first development is the standard we're building toward.

Teams were judged on three criteria:

  • Impact — Does this solve a real problem or meaningfully improve a process? Does it demonstrate genuine value that could carry into future work?
  • Execution — Is it functional and well-built? Did the team leverage AI tools effectively throughout development — not just at the end?
  • Presentation — Can the team clearly communicate their work? Judges evaluated lessons learned and workflow improvements that led to better outcomes.

What I Built

I built an MCP (Model Context Protocol) server for Core Data Platform.

If you're unfamiliar with MCP: it's a protocol that lets AI tools — Claude, other language models, or custom applications — communicate with external systems through a standardized interface. An MCP server exposes tools and resources; a client (Claude Code, in this case) calls them.

The server I built gives Claude direct, structured access to Core Data Platform capabilities. Instead of Claude having to ask users to click through a UI or run queries manually, it can interact with CDP programmatically — planning data extraction workflows, checking schema, validating configurations.

The intention is for Claude Chat users to interact with this MCP server through a Connector, unlocking Core Data Platform workflows directly within their chat experience. In the context of a hackathon, the value was immediate: Claude could actually reason about and operate on the platform in real time, rather than just describing what it would do.

Core Data Platform

Core Data Platform is a self-service ETL application that empowers users to extract datasets from the Solovis ecosystem.

Professionally, I've contributed to the platform across multiple layers:

  • API Services — Implemented RESTful endpoints using ASP.NET and Entity Framework to support key platform capabilities.
  • Infrastructure — Managed critical platform infrastructure in Microsoft Azure with Terraform.
  • Deployment — Authored and maintained CI/CD deployment code using Azure Pipelines for both the Core Data Platform API and UI codebases.
  • Frontend Integration — Integrated the Core Data Platform UI with backend service capabilities using TanStack Query for state management and API communication.

In short: it's a complex system with multiple moving parts — API, UI, infrastructure, data pipeline. An MCP server that exposes its functionality in a structured way makes it significantly more AI-friendly.

Building with Claude Code

This was the meta part of the hackathon: I used Claude Code to build the MCP server, while the MCP server itself was designed to work with Claude Code. AI all the way down.

The process mirrored the hackathon's own principles:

  • AI as Co-Pilot — Claude Code scaffolded the MCP server structure, suggested tool definitions, and helped me iterate on the API design. I didn't write it in isolation; it was a collaborative back-and-forth.
  • Prompt Engineering — The quality of the MCP server depended on clear, context-rich prompts about what Core Data Platform does, what operations Claude should be able to perform, and what the data contracts should look like.
  • Iterate with AI — I tested the server, found gaps, and Claude Code helped refine tool definitions and error handling. The feedback loop was fast.
  • Reflect & Improve — Along the way, I tracked what worked and what didn't — which tools were most useful, which API shapes made sense, which error cases needed explicit handling.

The Result

🏆 Second place!!!

One day — really just a few hours — to take an idea from concept to working demo in front of a panel. That was intense. The kind of intensity where you're in the flow, everything is happening at once, and you're not entirely sure if you're going to make it. But that's also what made it rewarding. You compress the entire feedback loop — plan, build, iterate, refine, demo — into a single afternoon.

And then there's the small miracle of demo gods being merciful. When you ready to present, the last thing you want to hear yourself say is "well, it was working a few moments ago." But this time, the demo worked. The MCP server responded, the tools fired, Claude reasoned about the platform. The relief was real. 😅

The broader lesson stuck with me: a new approach to building from nothing to quickly a proof-of-concept. Use AI at every stage. Write precise prompts. Review and refine quickly. Ship. Reflect on what worked.

That's the mindset worth carrying forward.
