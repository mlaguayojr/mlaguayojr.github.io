Recap

I was gifted an Anyuse eCalendar — a 16-inch Android touchscreen meant to serve as a household command center. Nearly every feature depends on WiFi, and the device drops its connection daily without recovering on its own.

My last attempt at a fix involved sideloading automation apps via ADB. I tried Tasker first, but it requires a paid license after the 7-day trial. I switched to Automate, built a flow to toggle WiFi on a timer, and hit a wall: toggling WiFi state requires root access, which I don’t have.

I closed that post having taken the “L” — stuck with what is, functionally, a very large digital photo frame. Read the full story in Part 1.

Diagnosis

Hardware debugging at this level — kernel drivers, WiFi HAL, ADB log analysis — is well outside my wheelhouse. So I pulled the device logs and handed them off to Claude to analyze.

The diagnosis came back with a clear answer: the WiFi scanning process is getting stuck. The driver enters a deadlocked state and, after sitting broken long enough, the device forces a disconnection rather than waiting any longer.

Suggestions

Before anything could be analyzed, the logs had to be collected. Claude outlined six commands to pull the relevant data from the device over ADB:

  1. WiFi Manager Logs — filters the main log buffer for anything WiFi-related. 
    adb logcat -b main -d "*:V" | grep -i "wifi" > wifi_logs.txt
  2. Complete System Log — a full logcat dump, ideally captured both before and after a disconnection. 
    adb logcat -d > full_logcat.txt
  3. WiFi Service State — a detailed dump of the WiFi service, connected networks, and driver info. 
    adb shell dumpsys wifi > wifi_dumpsys.txt
  4. Network Connectivity State — shows network transitions and connectivity changes. 
    adb shell dumpsys connectivity > connectivity_dumpsys.txt
  5. Kernel Logs — low-level output that can reveal WiFi driver or hardware faults. 
    adb shell dmesg > kernel_logs.txt
  6. Connected Network State — targeted slice of the WiFi dump focused on the active connection. 
    adb shell dumpsys wifi | grep -A 50 "mConnectedNetwork"


To catch the disconnection as it happens, Claude also suggested running continuous logging and stopping it manually once the device drops: 

adb logcat -b main > logcat_continuous.txt

Then search the captured output for keywords: disconnected, Disassociation, failed to authenticate, DHCP, signal loss, authentication failure.

Key Evidence

Four patterns stood out in the logs:

  1. Scanning deadlock — NL80211_CMD_TRIGGER_SCAN failed errors repeating continuously for roughly five minutes straight.
  2. Driver instability — log messages reading “Driver aic8800_fdrv not load? I’m not sure”, which is not exactly a confidence-inspiring thing to see in a system log.
  3. HAL communication failures — the WiFi HAL (Hardware Abstraction Layer) is unable to register its callbacks properly, cutting off the connection between the OS and the hardware.
  4. Forced disconnection — the device eventually initiates a disconnect with reason code 3, after the driver has been hung long enough that waiting is no longer an option.

The Culprit

The WiFi chip in this device is the AIC8800, made by Aicsemi, driven by aic8800_fdrv. The driver has a documented reputation for instability across platforms — its open issue tracker ↗ reflects ongoing compatibility problems, and a thread on the Armbian forums ↗ confirms it breaking across kernel versions even on actively maintained systems. This device runs Android 10, putting it in unsupported territory for a driver that struggles to keep up even on Linux.

Fixes to Try

Immediate Fixes (Try in Order)

  1. Update WiFi Driver. Check the current driver version, then look for firmware updates from the manufacturer. 
    adb shell modinfo aic8800_fdrv
  2. Factory Reset WiFi Settings. Clear the WiFi configuration cache and reboot. 
    adb shell rm -rf /data/misc/wifi/
adb shell pm clear com.android.wifi
adb reboot
  3. Check for System Updates. Go to Settings › About Phone › System Updates and install any pending updates, especially WiFi or modem firmware.
  4. Disable WiFi Power Saving. Android’s aggressive power management may be triggering the bug in the driver. On a plugged-in, wall-mounted device it serves no purpose. 
    adb shell settings put global wifi_scan_interval 60000
adb shell settings put global wifi_frequent_scan_interval 60000
  5. Reset Network Settings. Go to Settings › System › Reset and choose Reset WiFi, mobile & Bluetooth.

If Problem Persists

  1. Rollback Driver. Check if a previous firmware version is listed under Settings › About Phone › Build. The device manufacturer may provide a downgrade option.
  2. Contact Device Support. This appears to be a hardware/driver compatibility issue. The device may need warranty service or a driver patch from the manufacturer.

Outcome

Going through each suggestion honestly:

  1. Update the driver. Not something I’m comfortable doing. The risk of bricking the device outweighs the potential fix.
  2. Factory reset WiFi settings. The logs point to a hardware and driver issue, not stale configuration. I don’t think clearing a cache is going to fix this one.
  3. Check for system updates. Already on the latest version available. No help here.
  4. Disable WiFi power saving. Low enough risk that it’s worth trying. I’m skeptical, but I’ll give it a shot.
  5. Reset network settings. Same reasoning as #2 — this is a driver bug, not a configuration problem.
  6. Rollback the driver. Not possible without root or a flashable firmware package, neither of which I have.
  7. Contact device support. …sigh. I guess that’s where this ends up.

That said — while toggling WiFi state programmatically requires root, there’s another angle worth exploring: simulated taps. Instead of flipping the WiFi toggle directly, an automation flow could tap through the Settings UI to do it manually — no root required. Automate has an Interact Touch ↗ block that can send taps to arbitrary screen coordinates. If it can reach the WiFi toggle in Settings, the whole workaround from Part 1 might still be salvageable.
