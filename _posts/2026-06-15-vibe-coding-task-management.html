The Setup

No specs. No CLAUDE.md. No project instructions file. No carefully scaffolded prompts. Just me, Claude Code, and a blank directory.

The project: a task management system. Something concrete enough to build end-to-end, complex enough to stress-test the workflow. The repo is on GitHub: TaskManagementSystemVibe.

If you've read my post on the Solovis AI-First Hackathon, you already know I've used Claude Code to build things fast under pressure. That experience was structured — I knew the domain, I had a specific goal, and I had the context of work to anchor my prompts. This was different. This was pure vibe.

Why This Experiment

Scroll through any tech-adjacent feed — Facebook, Instagram, LinkedIn, the comment section of a news article about AI — and you'll find the same genre of post. Someone describing how vibe coding devolved into chaos: a tangled codebase, a model that kept rewriting everything, features that broke other features, sessions that lost all context, and a final product that technically ran but was impossible to reason about.

The warnings stack up. "It works until it doesn't." "You'll hit a wall around 1,000 lines." "The model starts hallucinating its own architecture." "You lose the thread and so does Claude."

I wanted to see it for myself. Not read about it, not watch a YouTube breakdown — actually sit down, open a terminal, and find out where the friction lives. A task management system felt like the right target: it's a domain everyone understands, has enough moving parts to be interesting (tasks, users, statuses, deadlines, maybe some persistence), and isn't so complex that failure would be trivially expected.

How It Went

The friction showed up in the details. Not in catastrophic failures, but in the kind of paper-cuts that add up as scope grows.

Database Migrations — Hit or Miss

Entity Framework migrations were about 50/50. Claude would generate migration code, but I'd run into a few instances where the .Designer.cs file went missing after applying the migration. The database itself was fine — Claude confirmed the schema changes were actually applied — but the project file stayed out of sync. To my half-surprise, Claude did sometimes suggest (and do) dropping the database entirely instead of addressing the migration itself.

Enum Drift

I'd define an enum with specific values — say, Created, InProgress, Completed. Claude would absorb that, but then when implementing logic that used the enum, it would use a similar value instead of the exact one: New instead of Created, for example. Between this and the Database Migrations, I had to frequently manually rebuild the application to catch build errors.

Feature Creep — And Feature Loss

I asked for a set of new features, or even to add on to existing logic. Somehow, some of those original features got stripped out. I had to go back, point out what was missing, and ask Claude to reincorporate them. This happened a few times. Each time, it felt like a context loss: the model wasn't tracking what we'd already built.

Implementing JWT - Namespace Hell

When I asked Claude to add JWT Authorization logic, it needed to find the correct namespace to work with. I'd set Claude to Auto mode with permission bypass (risky, I know), so it went into a loop spawning ILSpy processes to inspect DLLs and figure out what was available. My desktop turned into a gallery of ILSpy windows trying to answer the same question over and over. The approach worked eventually, but it wasn't elegant.

The Horrors — Real or Myth?

The hackathon gave me a reference point. During that build, I had context, structure, and a time box. This experiment stripped all of that away on purpose. The comparison is worth making: how much of what makes Claude Code effective comes from the structure you bring, versus what the tool provides on its own?

There's definitely friction in going in without a plan. Context loss, needing to /compact often, and needing to have Claude double-check its work are noticeable challenges in a vibe-coding session. But Claude did handle the mixed codebase of C# and React fairly well — so it's not that vibe coding is doomed, it's that structure matters more than I initially thought.

If I were to do this again, I'd introduce a few guardrails:

  1. Create a CLAUDE.md file. Include context about the project, domain, architectural patterns, and enum/naming conventions. This would have prevented enum drift and reduced Claude's second-guessing on what values to use.
  2. Implement features in isolated spec files. Define the feature, its inputs, outputs, and success criteria upfront. This creates a contract that Claude can reference without losing track of what was already built.
  3. Use separate chats per feature. This prevents context bloat and reduces the need to /compact. Each feature stays in its own thread, avoiding the cascading feature loss that happened when changes rippled across the codebase.
  4. Enforce a no-drop-database rule. Add explicit guidance that dropping the database should never happen without approval. Make it a conscious decision, not a shortcut Claude reaches for when migration debugging gets tedious.

Vibe coding isn't broken, but is definitely more successful and less painful when giving strong structure and guidance. The horrors people describe on social media should be a strong reminder to take that extra time to outline, document, poc, and ask questions. Structure first, vibes second 😎.
