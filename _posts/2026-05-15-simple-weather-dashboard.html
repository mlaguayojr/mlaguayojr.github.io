Why a Fixed-Location Dashboard?

Light Mode Dark Mode

Most weather apps let you search any city, swipe between multiple locations, and build a custom list. This one doesn't. It's intentionally simpler: you configure it once via environment variables (a ZIP code and country code), and it shows you the weather for that one place. No search box, no location history, no friction.

The idea is that you leave it open — in a browser tab, on a second monitor, on a tablet on your desk. It's a personal weather station that knows where you are and never bothers you to pick a place.

The Stack

The dashboard is built with a modern, deliberately chosen tech stack:

  • TanStack Start — React 19 with server-side rendering, not Next.js. It's newer, less mainstream, and worth the experiment.
  • TanStack Router — File-based routing that feels familiar if you've used Next.js.
  • TanStack Query — Data fetching, caching, and background refresh — more than a simple fetch call deserves.
  • Tailwind CSS v4 — The new major version with a different configuration model than v3.
  • Lucide React — Icon library for clean, minimal icons.
  • OpenWeatherMap API — Live weather data for the configured location.
  • Bruno — A local-first API client for testing endpoints. Newer than Postman, lighter than Insomnia.

How It Works

Geocoding the ZIP Code

The dashboard needs latitude and longitude to fetch weather data, but the configuration is a ZIP code and country code. A custom React hook handles the geocoding step: it calls OpenWeatherMap's free geocoding API to convert the ZIP code into coordinates. Once you have lat/lng, the weather hook takes over.

Fetching Weather Data

Another custom hook fetches the current weather from OpenWeatherMap using the coordinates. TanStack Query wraps both hooks, managing the async flow, caching responses, handling errors, and optionally refreshing in the background every 5 minutes.

The response includes everything: temperature, humidity, pressure, wind speed, visibility, cloud cover, and sunrise/sunset times. All of it gets displayed in the dashboard.

Components

The UI breaks into discrete, composable pieces:

WeatherDashboard (container)
  ├── WeatherHeader
  ├── MainWeatherCard
  ├── StatsGrid
  ├── WindSunriseSunset
  ├── ThemeToggle
  └── AutoRefresh

WeatherHeader shows the location and current date. MainWeatherCard is the hero section with the large temperature and condition. StatsGrid lays out humidity, pressure, and visibility in a responsive grid. WindSunriseSunset displays wind direction/speed and solar times.

ThemeToggle and AutoRefresh are the interactive bits — toggle dark mode and set up background refresh, respectively.

Dark Mode Done Right

The dashboard supports light and dark themes. Rather than relying only on Tailwind's CSS class toggle, a centralized ThemeContext manages the state in React. This approach buys you:

  • Consistent theme propagation across the component tree — no surprise unstyled flashes.
  • The ability to persist the user's choice in localStorage and restore it on reload.
  • Easy integration with component-level logic that might respond to the theme (e.g., adjusting API calls or image sources).

Tailwind still does the heavy lifting with variant utilities, but the React context handles the orchestration.

Auto-Refresh with TanStack Query

One of the reasons to use a full data-fetching library like TanStack Query: an optional auto-refresh feature that refetches the weather every 5 minutes. A single refetchInterval config flag gets you:

  • Automatic background refresh without writing an interval manager yourself.
  • Exponential backoff and retry logic if the API is temporarily down.
  • Stale-data handling — TanStack Query knows whether the cached data is fresh or stale.
  • No extra boilerplate, no cleanup hassles.

For a simple app that just needs to refresh the weather once in a while, it might seem overkill. But once you've used TanStack Query, you realize how much complexity it absorbs for you.
